# Security Audit — 2026-04-16

> **Auditor**: Claude Code (pre-v1.0 release audit)
> **Scope**: `src/nous.cyr` (2,400+ lines), all public API functions
> **Version**: 0.3.0 (pre-release)

---

## Internal Audit Findings

### P0 — Critical

**1. Command Injection in sysdb_* functions** — FIXED
- **Location**: `sysdb_search`, `sysdb_is_installed`, `sysdb_get_installed`, `sysdb_info`
- **Issue**: Package names and queries were interpolated into shell command strings via `cmd_run()`, allowing injection via crafted names like `"foo; rm -rf /"`.
- **Fix**: Replaced shell string concatenation with array-based argument passing via `exec_capture(args_vec, ...)`. Each argument is passed as a separate execve argv entry. Added `validate_package_name()` check at entry to each sysdb function.

**2. NULL Dereference in accessor chains** — FIXED
- **Location**: `mpkg_name`, `mpkg_version`, `mpkg_desc`, `mpkg_runtime`, `mpkg_deps`
- **Issue**: Chained `load64()` calls (e.g., `ai_name(mf_agent(mp_manifest(p)))`) would crash if any intermediate pointer was NULL.
- **Fix**: Added NULL checks at every dereference level. Safe return values (0 or empty defaults) on NULL.

### P1 — High

**3. Unchecked return values** — MITIGATED
- **Location**: `payload()` calls after `is_ok()` checks
- **Status**: All `payload()` calls are preceded by `is_ok()` or `is_err_result()` checks. Callers that use `payload()` on vec results check `vec_len() > 0` before accessing elements. Risk is mitigated but not fully eliminated for edge cases.

**4. Path Traversal in scan_installed** — FIXED
- **Location**: `scan_installed()` directory entry processing
- **Issue**: Directory entries from `dir_list()` were used directly in `join_path()` without validation. Entries like `"../../etc"` could escape the registry directory.
- **Fix**: Added explicit rejection of entries containing `".."` or `"/"` before path construction.

**5. Integer Overflow in semver_parse** — FIXED
- **Location**: `semver_parse()` digit accumulation loops
- **Issue**: Unbounded `major = major * 10 + digit` could overflow i64, wrapping to small values and bypassing version constraints.
- **Fix**: Added `MAX_SAFE` bound check (INT64_MAX / 10) before each multiplication. Returns 0 on overflow.

### P2 — Medium

**6. TOCTOU in scan_installed** — FIXED
- **Location**: `scan_installed()` file_exists + file_read_all sequence
- **Issue**: Race window between checking file existence and reading the file.
- **Fix**: Removed `file_exists()` check. `parse_manifest_file()` already returns 0 on read failure.

**7. Fixed-size buffers** — ACCEPTED RISK
- **Location**: `cmd_buf` (64KB), manifest read buffer (8KB), recipe parse buffer (64KB)
- **Issue**: Commands or files exceeding buffer size are truncated.
- **Status**: `exec_capture` and `file_read_all` respect buffer size limits. Truncation causes parse failure (returns 0/empty), not buffer overflow. Acceptable for current use cases. Document size limits.

**8. Forward slash in package name validation** — ACCEPTED RISK
- **Location**: `is_valid_name_char()` allows `/` (ASCII 47)
- **Issue**: Package names like `"publisher/name"` are valid (marketplace format). Could be used in path construction.
- **Status**: Path traversal is blocked in `scan_installed()` by the `..` and `/` check on directory entries. Package names are validated before shell execution. The `/` is required for the marketplace publisher/name format. Risk is mitigated.

## External Research — Known Attack Patterns

### Dependency Confusion — HIGH RISK
- **CVE-2021-23135** (npm/pip namespace confusion)
- **Nous exposure**: `MarketplaceFirst` strategy means marketplace packages shadow system packages of the same name. An attacker who publishes `openssl` to the marketplace would override the system `openssl`.
- **Defense**: Require `publisher/name` format for marketplace. Warn on cross-source name collisions. Consider `--deny-source-override` flag.

### Manifest Injection — MEDIUM RISK
- **CVE-2022-29244** (npm), **CVE-2023-26136** (tough-cookie)
- **Nous exposure**: No field length limits on manifest JSON. A 2GB description field would exhaust memory.
- **Defense**: Cap manifest file size (already 8KB buffer limit). Add field length validation.

### Typosquatting — LOW-MEDIUM RISK
- **CVE-2017-16138** (npm `crossenv`), hundreds of PyPI incidents
- **Nous exposure**: `suggest_package()` helps users find the right name but doesn't prevent installation of typosquatted packages that exist.
- **Defense**: Require `publisher/name` namespacing. Block names within edit distance 1 of popular packages at publish time (mela responsibility).

### Lockfile Poisoning — NOT APPLICABLE (YET)
- Nous has no lockfile mechanism. When added, include SHA-256 hashes and source URLs per package.

### ReDoS in Version Parsing — NOT VULNERABLE
- **CVE-2022-25883** (semver npm package)
- **Nous**: Hand-written character-by-character parser, no regex. Linear time. Not vulnerable.

### Unsigned Package Acceptance — HIGH RISK
- **CVE-2019-3462** (APT signature bypass), **CVE-2016-1252** (APT InRelease bypass)
- **Nous exposure**: `trusted` field is hard-coded to `true`. No actual signature verification. Marketplace has no signing infrastructure.
- **Defense**: Implement package signing (sigil integration). Never hard-code trust. Add `--require-signatures` flag.

### Path Traversal — LOW RISK (FIXED)
- **CVE-2018-1002200** (Zip Slip)
- **Nous**: `scan_installed()` now rejects `..` and `/` in directory entries. Not vulnerable for manifest scanning. Must be re-checked when `install_package()` is implemented for real.

### Supply Chain Attacks — MEDIUM-HIGH RISK
- **event-stream** (2018), **ua-parser-js** (2021), **xz-utils CVE-2024-3094** (2024)
- **Nous exposure**: No review process for marketplace. Community recipes are un-audited. `MarketplaceFirst` strategy means compromised marketplace packages take precedence.
- **Defense**: Package pinning with hashes. Multi-party signing for popular packages. Version update warnings. Quarantine period for new versions.
